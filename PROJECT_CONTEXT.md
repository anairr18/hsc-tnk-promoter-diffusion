# DNA-Diffusion HSC Promoter Project — Context Handoff

This document summarizes the full project history so work can resume in a
new environment (e.g. Codex) without re-deriving prior decisions.

## Project Goal

Visiting student project at Harvard Medical School (Pinello Lab). Goal: use
DNA-Diffusion to generate synthetic, cell-type-specific promoter/enhancer
sequences, validate them computationally with Enformer, and eventually test
top candidates in a wet-lab MPRA (massively parallel reporter assay). Long
term target is hematopoietic stem cells (HSCs); current phase uses cultured
cell lines (K562, HepG2, GM12878) as a faster proof of concept before moving
to the more complex hematopoietic cell types (T, NK, B cells, HSCs).

Advisor: Giacomo. All dataset design choices below were made through several
rounds of his feedback.

## Why Codex Instead of Colab

Colab sessions were disconnecting / hanging during long unattended training
runs, and subprocess output was buffered so progress was invisible
("Launching training..." with no further output even though training was
likely running). Moving to a more controllable environment to run training
end-to-end with visible logs and resumability.

## Key Files in This Bundle

### `pipeline/cell1_dataset_and_training.py`
Single script that: mounts/uses local cache dir, downloads and processes
GENCODE genes, RNA-seq (ENCODE), DNase-seq peaks (ENCODE), hg38 genome,
builds the final 9,696-sequence dataset per the parameters below, clones and
sets up DNA-Diffusion via `uv`, reformats data with the required header row,
and launches `train.py`. This was written for Colab (`google.colab.drive`)
— if running outside Colab, the `drive.mount(...)` call and
`/content/drive/MyDrive/...` paths need to be replaced with a local/cloud
path of your choosing. Everything else (download logic, processing, model
setup, training command) is environment-agnostic.

### `pipeline/cell2_generate_and_backup.py`
Run after training completes (or has at least one checkpoint). Finds the
most recent checkpoint, runs `sample.py` to generate 1000 sequences per cell
type, and backs up checkpoints/logs/generated sequences. Also written for
Colab — same caveat about `drive.mount`.

### `pipeline/hsc_complete_pipeline.py`
Earlier combined single-cell version (dataset + training + generation +
backup all in one blocking script). Superseded by the cell1/cell2 split,
kept here for reference only — prefer the split version since a single
giant blocking call loses checkpoint backup if the run dies mid-training.

## Dataset Pipeline — Final Parameters (Giacomo-approved)

All of these came out of an iterative feedback loop. Final state:

| Parameter | Value | Rationale |
|---|---|---|
| Conditioning | 3-way: K562, HepG2, GM12878 (no CELLTYPE- groups) | Giacomo: don't need to train the model to generate sequences that are open-but-inactive in the target cell type |
| RNA-seq datasets per cell type | 2 (ENCODE experiments, listed below) | Sufficient for cultured cell lines; more datasets needed later for hematopoietic cells |
| TPM aggregation | **Median**, not mean | More robust to outlier TPM values; mean gave unreasonably high fold-change ratios (296x–1113x) |
| Differential expression threshold | ratio > 2.0 AND TPM(target) > 1.0 | Giacomo confirmed this threshold is fine |
| DNase peaks per cell type | Top 50,000 **by accessibility score** (not random sample) | Random sampling was wasteful since most peaks get excluded anyway during gene-linking; selecting top-scoring peaks first preserves more usable signal |
| Peak-gene linking window | 10kb (symmetric, centered on peak vs TSS) | Acceptable for now; could later move to asymmetric `[TSS-50k, TSS+10k]` (strand-aware) for hematopoietic cells, but added complexity isn't needed yet |
| Sequence extraction | 200bp windows; stride 50bp for peaks >200bp (was 100bp, reduced per Giacomo: "1/4th of the target length") | More sequences per long peak without too much redundancy |
| Class imbalance | Accepted, no balancing | Giacomo: not a concern for this evaluation phase; hematopoietic cell types (T/NK/B) should have more balanced data quality across experiments |
| Train/val/test split | Random stratified 80/10/10 | Giacomo preferred this over the earlier chromosome-based split (chr1=test, chr2=val), which produced too-small val/test sets |

## Dataset Evolution / Iteration History (for context)

1. **First attempt**: 8+ hour runtime due to naive nested-loop peak-gene
   linking (checking every peak against every gene = billions of
   comparisons). Fixed with chromosome pre-filtering + peak sampling →
   reduced to ~30 min, then ~2-3 min with caching.
2. **First "fast" pipeline**: 10,549 sequences, 3-way conditioning, but no
   differential expression filtering (linked peaks to ALL genes, not just
   cell-type-specific ones) — flagged as lacking biological precision.
3. **First "complete" pipeline (Giacomo's initial feedback round)**:
   implemented multi-dataset RNA-seq merging (mean TPM), differential
   expression filtering, 6-way conditioning (CELLTYPE+/-). Produced 3,579
   sequences. Differential expression ratios were very high (296x-1113x)
   due to using mean instead of median.
4. **Giacomo's final feedback round** (the 6 points in the table above)
   → produced the current 9,696-sequence dataset described above.

## DNA-Diffusion Setup Notes (Important Gotchas)

- **Package manager is `uv`, not pip.** `pip install hydra-core` alone is
  NOT sufficient and will miss other dependencies — use `uv sync` after
  cloning the repo.
- **Training command** (confirmed working, no Hydra override errors):
  ```bash
  uv run train.py \
      data.data_path=data/train.txt \
      data.load_saved_data=False \
      data.saved_data_path=data/hsc_encode_data.pkl \
      training.use_wandb=False
  ```
- There is **no `train.checkpoint_path` key** in the Hydra config — early
  attempts to override this failed with `Key 'train' is not in struct`.
  The actual config structure (from `configs/data/default.yaml` and
  `train.py` inspection) only exposes `data.*`, `model.*`, `optimizer.*`,
  `diffusion.*`, and `training.*` namespaces. Fine-tuning from the
  HuggingFace pretrained checkpoint (`ssenan/DNA-Diffusion`) was NOT
  confirmed to work through `train.py` — that checkpoint appears intended
  for `sample_hf.py` (sampling only), not as a resume-from point for
  `train.py`. Current approach trains from scratch on our dataset, which
  Giacomo's feedback frames as acceptable ("for the first training we don't
  need [more data], especially because it is a finetuning of a model that
  has already been trained" — this framing assumes fine-tuning, but the
  technical mechanism for loading pretrained weights into `train.py` was
  never fully resolved. **This is an open question — worth investigating
  in the new environment**, possibly by inspecting `src/dnadiffusion/`
  model loading code more closely, or asking Giacomo directly whether
  training-from-scratch-on-small-data vs true checkpoint-resume matters for
  this phase.)
- **Data format**: must include header row with exact column names `chr`,
  `sequence`, `TAG` (tab-separated). Files saved without headers caused
  `KeyError('chr')` in the dataloader.
- Required full config example (from `configs/train/default.yaml`
  inspection, embedded in train.py's hydra config):
  ```yaml
  model:
    _target_: src.dnadiffusion.models.unet.UNet
    dim: 200
    channels: 1
    dim_mults: [1, 2, 4]
    resnet_block_groups: 4
  data:
    _target_: src.dnadiffusion.data.dataloader.get_dataset
    data_path: data/train.txt
    saved_data_path: data/hsc_encode_data.pkl
    load_saved_data: false
    debug: false
  optimizer:
    _target_: torch.optim.Adam
    lr: 0.0001
  diffusion:
    _target_: src.dnadiffusion.models.diffusion.Diffusion
    timesteps: 50
    beta_start: 0.0001
    beta_end: 0.2
  training:
    distributed: false
    precision: bf16
    num_workers: 2
    pin_memory: true
    batch_size: 120
    sample_batch_size: 10
    num_epochs: 5000
    min_epochs: 2000
    patience: 10
    log_step: 50
    sample_epoch: 500
    number_of_samples: 1000
    use_wandb: true   # set to false via override
  ```

## Timing Estimates (Unverified — Watch First Few Epochs)

The model is small (UNet, dim=200, 3 resolution levels, 1 channel, 50
diffusion timesteps) relative to typical image-diffusion models. With 7,757
training sequences and batch_size=120 (~65 batches/epoch), rough estimate on
an A100 was 2-6 hours for the full 5000 epochs, but this was NOT measured —
only estimated from architecture size. `min_epochs=2000` + `patience=10`
means early stopping is possible once validation loss plateaus. Recommend
checking actual seconds-per-epoch from the first 50-100 epochs of real logs
before assuming a total runtime.
